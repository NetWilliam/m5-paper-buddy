# ESP32-S3-RLCD-4.2 屏幕旋转与驱动适配指南

> **目标**：将 400x300 (Landscape) 的 ST7305 反射屏适配为 300x400 (Portrait) 竖屏显示。

## 1. 硬件引脚陷阱 (Pin Mapping)
这款硬件有多个变体。微雪 Wiki 标注的引脚在某些板子上是无效的。经过验证，**能够成功读写**的配置如下：

| 功能 | 引脚 (GPIO) | 备注 |
| :--- | :--- | :--- |
| **LCD_CS** | **40** | 关键！不是 10 |
| **LCD_DC** | **5** | 关键！不是 17 |
| **LCD_RST** | **41** | 关键！不是 21 |
| **LCD_SCK** | **11** | SPI3 / SPI2 通用 |
| **LCD_MOSI** | **12** | SPI3 / SPI2 通用 |
| **LCD_HOST** | **SPI3_HOST** | (Host ID: 2) |

## 2. 旋转逻辑：硬件 vs 软件
**核心痛点**：ST7305 使用的是一种特殊的 **2x4 像素块打包存储** 机制。
- **硬件旋转 (MADCTL)**：如果通过 `0x36` 寄存器设置硬件旋转，会破坏 2x4 的内存映射，导致屏幕出现结构性的条纹乱码。
- **解决方案**：保持硬件在物理 **400x300 (Landscape)** 模式。在驱动层拦截写像素操作，通过数学计算进行 **软件顺时针 90 度旋转**。

### 软件旋转公式 (300x400 Portrait -> 400x300 Physical)
```cpp
// 目标：将 LVGL 传来的 (x, y) 映射到 物理坐标 (phys_x, phys_y)
// x 范围 [0, 299], y 范围 [0, 399]
uint16_t phys_x = (400 - 1) - y;
uint16_t phys_y = x;
```

## 3. 驱动稳定性优化
### 移除 BUSY 引脚检测
- **问题**：部分 RLCD 板子的 BUSY (GPIO 47) 信号在初始化时会一直拉高，导致驱动阻塞（每条指令超时 200ms，总初始化长达 20 秒）。
- **优化**：在 `WaitBusy()` 中做 no-op 处理。改用 `vTaskDelay`。

### 初始化时序
- **Reset**：必须保证低电平持续至少 50ms，恢复后等待 150ms 以上。
- **窗口设置**：ST7305 有内部偏移。物理窗口设置为 `0x2A: [0x12, 0x2A]` (Column) 和 `0x2B: [0x00, 0xC7]` (Row) 才能对齐 400x300。

## 4. 软件层配置
### LVGL 配置
- `lv_display_create` 应该申报为 **300x400**。
- 使用 **RGB565** 作为中间缓冲区（LVGL 绘制），在 `FlushCb` 中将其转换为 1-bit 并应用上述旋转逻辑。

### 内存管理
- 必须确保 `disp_buffer` 和 LVGL `draw_buf` 分配在 **MALLOC_CAP_SPIRAM** (PSRAM) 中，否则 300x400 的双缓冲可能会撑爆内部 SRAM。

## 5. 编译建议
- 修改 `config.h` 中的引脚定义后，务必执行 `idf.py fullclean`。ESP-IDF 的增量编译有时无法正确识别跨文件的宏定义更改。
